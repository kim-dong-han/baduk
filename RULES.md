# RULES

> 모든 세션에 적용. 목적: 토큰 최소화 · 컨텍스트 오염 방지 · 스타일 일관성.

## Claude 응답 규칙 (토큰 최소화 — 최우선)
1. **변경된 부분만 출력.** 전체 파일 재출력 금지. Edit 도구로 diff만.
2. 설명은 기본 **3줄 이내**. 요청 없으면 배경/대안 나열 금지.
3. 끝나면 한 줄 요약(무엇을 바꿨나)만. 코드 재인용·자기칭찬 금지.
4. 모른다/확인 필요는 즉시 말한다. 추측으로 코드 만들지 않는다.
5. 작업 전 **계획을 장황하게 적지 않는다.** 바로 실행, 결과만 보고.
6. 파일 위치는 코드 탐색 전 **ARCHITECTURE.md** 우선 확인.
7. 한 번에 끝낼 일은 한 응답에. 불필요한 되묻기·중간 확인 금지(아래 "확인이 필요한 경우" 제외).

## 컨텍스트 오염 방지
- 무관한 파일 읽지 않기. 필요한 범위만 Read(offset/limit).
- 큰 탐색은 결론만 필요할 때 Explore 에이전트 사용 후 요약만 수용.
- 사실은 문서(PROJECT/ARCHITECTURE)에 고정. 세션마다 재발견 금지.

## 개발 규칙
- 기능 완료 시 즉시 commit (+ 사용자 요청 시 push). 작은 단위 커밋.
- 구조 변경 시 ARCHITECTURE.md, 상태 변경 시 TODO.md 동시 갱신.
- 설정값(경로/타임아웃/모델)은 `application.yaml`에만. 하드코딩 금지.
- 검증 안 한 변경은 "미검증"이라고 명시.

## 코드 스타일
- Java 21, 4-space 인덴트, Lombok `@Builder`/`@Getter` 적극 사용.
- DTO는 builder 패턴, 불변 지향. 생성자 주입(@Autowired 필드주입 금지).
- 좌표 변환은 항상 `CoordinateConverter` 경유 (직접 변환 금지).
- 로그는 기존 `[태그] 메시지` 형식 유지 (예: `[SingleGame]`).
- 주변 코드 컨벤션을 따른다. 새 라이브러리 임의 도입 금지.

## 사용 기술 제한
- 신규 의존성 추가 금지(요청·승인 없이). build.gradle 목록이 전부.
- DB/ORM 도입 금지 — 영속은 파일(JSON) 유지.
- 프런트: Thymeleaf + 바닐라 JS + Chart.js. 프레임워크(React 등) 금지.
- Thymeleaf 3.1 인라인 표현식 제약 준수.

## 출력/UI 규칙
- 지표는 일반인 친화적 한국어 (등급: 최선/좋음/보통/실수/악수).
- 모호한 지표·축 도입 금지. 직관적 Y축.
- 배지 색: 초록(최선·좋음)→앰버(보통)→주황(실수)→빨강(악수).

## 확인이 필요한 경우 (이때만 되묻는다)
- 되돌리기 어려운 작업(force push, 파일 대량 삭제, 히스토리 변경).
- 외부로 나가는 작업(원격 push, 배포) — 단 push는 훅 자동 실행.
- 요구가 코드/문서/기본값으로 해결 불가할 때만 질문.

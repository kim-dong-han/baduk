# PROMPTS

> 복사해서 붙여넣는 요청 템플릿. `{}` 만 채운다.
> 모든 템플릿은 RULES.md(변경분만 출력·설명 최소)를 전제로 한다 — 매번 다시 적지 말 것.
> 공통 머리말(생략 가능, 이미 RULES에 있음): "변경된 부분만. 설명 3줄 이내."

## 기능 추가
```
기능: {무엇을}
위치: {화면/엔드포인트, 모르면 ARCHITECTURE 참고해 결정}
조건: {제약·엣지케이스}
완료 후: 커밋. ARCHITECTURE/TODO 영향 있으면 같이 갱신.
```

## 버그 수정
```
증상: {현상}
재현: {입력/경로}
기대: {정상 동작}
원인 추정 후 변경분만 수정. 추측 금지, 불명확하면 먼저 확인.
```

## 리팩토링
```
대상: {파일/메서드}
목표: {중복 제거 / 가독성 / 분리}
제약: 동작 변경 금지. public 시그니처 유지. 변경분만 diff.
검증: 빌드 통과 여부 보고(미검증이면 명시).
```

## API 생성
```
엔드포인트: {METHOD /api/...}
요청/응답: {DTO 필드}
규칙: Controller는 매핑만, 로직은 Service. DTO는 builder.
      기존 SingleGameController 패턴 따름. 변경분만.
```

## 테스트 작성
```
대상: {클래스/메서드}
케이스: {정상 / 경계 / 실패}
규칙: JUnit5. 외부(KataGo/파일)는 모킹 또는 fixture.
      기존 test 패키지 컨벤션 따름. 변경분만.
```

## DB 변경 (= JSON 스키마/저장 구조 변경)
```
대상: {DTO 필드 / 결과 JSON 구조 / 저장 경로}
변경: {추가·삭제·이름}
주의: 기존 저장 JSON 하위호환. 깨지면 "재분석 필요" 명시.
      application.yaml 경로 영향 확인. ARCHITECTURE "DB 구조" 갱신.
```

## 문서/상태 갱신
```
방금 작업 반영해 TODO.md만 갱신. (구조 바뀌었으면 ARCHITECTURE도)
설명 없이 변경분만.
```
